Remarks
The compoentent loader takes two kinds of inputs, types of objects to create and already created auxilliary objects seeding the dependency injection process.

For each type of an object to create, the instantiation filter attributes implementing IComponentInstantiationFilter are evaluated. If any of them prohibits the creation of an object, the type will be discarded.

The remaining types will be created using a two-phase model. In the first phase, all objects having a constructor whichs arguments can be perfectly bound (TrySelectPerfectFitT(IEnumerableT, T)) will be instantiated. If no type with a perfectly bindable constructor is left, the second binding phase will be initiated. This phase uses approximate fitting (TrySelectAcceptableFitT(IEnumerableT, T)) to bind the arguments of the constructors. When it successfully bound all parameters of a constructor, an object will be created which then will perticipate in another first phase binding atempt.

Constructor parameters can be tagged with attributes like OptionalAttribute to make a second phase binding possible. Untagged parameters will be bound to an auxilliary object or a previously created object that can be assigned to it, thus are always perfectly fitting.

To resolve circular dependencies, one of the objects has to implement a ICircularDependencyT interface for the specific type. Its ResolveDependency(T) method will be invoked after all objects have been created, using the same fitting mechanisms as decsribed for the constructor parameters.

If a list of all objects of a given interface or type is required, the ICollectionDependencyT attribute should be used.

When an objects requires facilities to dynamically instantiate objects using the described dependency resolution process, it should implement IFactoryDependency.
